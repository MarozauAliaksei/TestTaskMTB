## Архитектурные решения

- **LLM**: Ollama + `qwen2.5:3b-instruct`, локально на GPU — без внешних API-ключей
  и рейт-лимитов на демо. Модель уменьшена с 7B до 3B под конкретное железо
  (GTX 1060, 6GB VRAM): Whisper и LLM должны одновременно помещаться в память,
  7B туда не влезала бы с запасом вместе с ASR-моделью.
- **ASR**: faster-whisper `small` (не `medium`) — по той же причине, компромисс
  между точностью и VRAM. Проверяется через WER-таблицу (см. ниже) — если
  качества не хватит, вернёмся к `medium`.
- **Диаризация**: упрощённая, по реальным паузам в аудио — но НЕ по тайм-кодам
  сегментов Whisper (это не сработало на практике, см. ниже), а через прямой
  анализ амплитуды исходного файла (`pydub.silence.detect_nonsilent`).
  Найденные "острова" речи чередуются Оператор/Клиент/Оператор/... — это всё
  ещё не настоящая диаризация (не различает голоса), но привязана к реальным
  паузам в записи, а не к произвольной сегментации Whisper.

  **Почему не тайм-коды Whisper** — два реальных прогона на практике показали,
  что на них нельзя полагаться: с `vad_filter=True` Whisper физически вырезает
  тишину и сшивает тайм-коды сегментов встык (паузы теряются полностью); с
  `vad_filter=False` Whisper всё равно режет текст на сегменты по своей
  внутренней логике (где заканчивается "предложение"), а не по факту тишины —
  в обоих случаях реальные 700мс паузы между репликами в тайм-кодах не
  отражались надёжно. Отсюда и решение искать паузы независимо, напрямую по
  сигналу.

  **Третья находка, тоже с реального прогона:** переход на "острова по
  амплитуде" сам по себе оказался недостаточным — `edge-tts` вставляет
  естественные паузы на запятых даже ВНУТРИ одной непрерывной фразы одного
  говорящего, и длительность этих пауз плывёт настолько, что фиксированный
  порог тишины (что 500мс, что 900мс) не мог надёжно отличить их от паузы на
  смене говорящего.

  **Финальное решение:** вместо фиксированного порога — адаптивный, по
  "естественному разрыву" в распределении пауз конкретного файла. Сканируем
  с низким порогом (300мс), ловим вообще все паузы, затем сортируем их длины
  и ищем, где между соседними по величине значениями самый большой скачок —
  это и есть граница "короткая пауза / реальная смена говорящего" именно для
  этого файла, не универсальная константа. (Первая версия этого подхода
  использовала медиану вместо разрыва — оказалась статистически неверной для
  диалогов с частой сменой говорящего, где сама медиана уже "большая";
  поймано локальным тестом на синтетических данных до реального прогона.)

  **Известное ограничение, зафиксированное честно:** если в файле вообще нет
  внутрифразовых пауз (все паузы одинаковой длины — унимодальное
  распределение), метод может по ошибке склеить одну реальную смену
  говорящего. Для наших тестовых диалогов маловероятно (почти в каждой фразе
  есть запятая), но не исключено на отдельных коротких репликах. Дальнейшее
  уточнение уже не даст пропорциональной отдачи — диаризация в ТЗ явно
  обозначена как "базовая", настоящая диаризация (`pyannote.audio`) —
  бонусная задача.
- **Оркестрация агентов**: LangGraph, простой DAG — classifier/quality/compliance
  запускаются параллельно (не зависят друг от друга, только от транскрипта),
  summarizer стартует после того, как все три завершились. Supervisor-паттерн
  не используется: агенты не принимают решений о маршрутизации друг друга —
  честнее показать это как pipeline, а не как "автономных агентов".
- **Передача аудио в чат**: через ссылку (URL) в сообщении, а не через
  вложение файла. У доступа к файлам-вложениям в OpenWebUI Pipelines сейчас
  есть известные нерешённые проблемы в некоторых версиях — URL надёжнее и
  работает одинаково что в чате, что в `/analyze`.

## Быстрый старт

```bash
cp .env.example .env
sudo docker compose up -d --build
sudo docker exec -it mtbank-ollama ollama pull qwen2.5:3b-instruct
```

Дальше — либо через чат OpenWebUI (`http://localhost:3000`, нужно один раз
подключить Pipelines в Settings → Admin Settings → Connections, адрес
`http://pipelines:9099`, ключ — значение `PIPELINES_API_KEY` из `.env`),
либо напрямую через API:

```bash
curl -X POST http://localhost:8000/analyze -F "file=@test_data/call_1_kredity.mp3"
```

## Тестовые данные и WER

**1. Сгенерировать (на хосте, нужен интернет для edge-tts):**
```bash
pip install edge-tts pydub
sudo apt-get install ffmpeg   # если ещё не стоит на хосте
python3 test_data/generate_test_data.py
```
Создаёт 5 диалогов оператор/клиент (кредиты, карты, переводы, жалобы, другое)
+ одну 8kHz-версию (телефонное качество) + эталонные транскрипты (`.txt`)
рядом с каждым аудио. В конце скрипт сам считает и выводит суммарную
длительность — если меньше 5 минут, допиши реплик в `DIALOGUES` внутри
скрипта и перезапусти.

**2. Посчитать WER (внутри контейнера, там уже есть Whisper + GPU):**
```bash
sudo docker compose exec pipelines python3 test_data/compute_wer.py
```

**Таблица WER** (реальный прогон, `faster-whisper small`, GPU):

| Файл | Сегментов | WER |
|---|---|---|
| call_1_kredity.mp3 | 19 | 0.060 |
| call_1_kredity_8khz.wav | 20 | 0.053 |
| call_2_karty.mp3 | 18 | 0.062 |
| call_3_perevody.mp3 | 12 | 0.165 |
| call_4_zhaloby.mp3 | 15 | 0.067 |
| call_5_drugoe.mp3 | 9 | 0.274 |

**Методологическая оговорка:** эталонные транскрипты написаны словами
("двенадцать процентов", "триста тысяч"), а Whisper часто распознаёт числа
цифрами ("12%", "300 тысяч") — для `jiwer` это разные токены, хотя по смыслу
идентично. Это, вероятнее всего, основная причина повышенного WER у
`call_3_perevody` и `call_5_drugoe` — в них больше числительных и указаний
времени, а не признак худшего качества распознавания именно на этих файлах.
Честного посимвольного нормализованного сравнения (приведение чисел к одному
формату перед WER) в `compute_wer.py` пока нет — это было бы более точной
метрикой, но и так видно, что качество распознавания в целом хорошее (5 из 6
файлов — WER < 0.07).

**GPU-находка, тоже с реального прогона:** faster-whisper всё это время
работал не на GPU, а откатывался на CPU — `ctranslate2` (движок faster-whisper)
корректно определял, что GTX 1060 (архитектура Pascal) не имеет эффективных
fp16-вычислений (это появилось только с Volta+), и отказывался считать в
`float16`. Исправлено: `WHISPER_COMPUTE_TYPE` теперь `int8_float16` —
стандартная рекомендация `ctranslate2` именно для GPU без быстрого fp16,
использует int8-веса с float16-накоплением, хорошо поддерживается на Pascal.

## Деплой (Railway)

Публичное демо развёрнуто без self-hosted Ollama — вместо неё используется
Groq (бесплатный tier, вариант прямо разрешён в FAQ ТЗ: *"Если нет GPU —
Groq/Together/OpenRouter дают бесплатный tier"*). Локально по-прежнему Ollama,
дублировать инфраструктуру ради демо-хостинга избыточно.

Три сервиса вместо четырёх (без `ollama`):

| Сервис | Источник | Dockerfile |
|---|---|---|
| `pipelines` | этот репозиторий | `Dockerfile.pipelines` |
| `api` | этот репозиторий | `Dockerfile.api` |
| `open-webui` | `ghcr.io/open-webui/open-webui:main` | — (готовый образ) |

Переменные окружения `pipelines`/`api` на Railway:
```
LLM_BASE_URL=https://api.groq.com/openai/v1
LLM_MODEL=<актуальная модель из консоли Groq>
LLM_API_KEY=<реальный ключ Groq>
WHISPER_MODEL=small
WHISPER_DEVICE=cpu
WHISPER_COMPUTE_TYPE=int8
```

`WHISPER_DEVICE=cpu` — на Railway нет гарантированного GPU. CPU уже проверен
локально (39.5с на 96-секундный файл) — с запасом укладывается в лимит 60с.

**Находка с реального деплоя:** на лимите памяти Railway (1GB на используемом
плане) модель `small` не поместилась — процесс убивался OOM-киллером
посреди обработки (`Killed` в логах сразу после начала транскрибации).
Для облачного деплоя используем `WHISPER_MODEL=base` вместо `small` —
заметно меньше памяти, локальная версия (и WER-таблица выше) при этом
остаётся на `small`, для облака это отдельный, осознанный компромисс.
